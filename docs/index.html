Athlynk
Codex.

Riferimento tecnico della piattaforma Athlynk — l'API mobile Django e l'app atleta SwiftUI. Strutturato come un reference di simboli: moduli, endpoint, modelli e viste, ognuno con firma, scopo e contratto.

Panoramica

Il sistema

Athlynk collega atleti e professionisti (coach, trainer, nutrizionisti). Esistono due frontend sullo stesso dominio Django: la web app (coach + atleta) e l'app iOS nativa dedicata all'atleta. L'app iOS non parla mai col database direttamente: consuma una proiezione JSON di sola-lettura-prevalente esposta sotto /api/v1.

Backend

Django 6 · custom-auth · Postgres (Supabase) · cache DB per rate-limit. Nessuna dipendenza DRF: viste semplici che ritornano JsonResponse.

Mobile

SwiftUI · async/await · token in Keychain · estetica greca-lusso (Didot, pergamena, bronzo/Egeo).

Confine

35 endpoint /api/v1. Auth a token firmati via header Authorization: Bearer.

Ciclo di vita della richiesta

1 · Middleware
Security, Session, CSRF, Auth, Clickjacking, SanitizationMiddleware (sanitizzazione input).
2 · api_view
Esente CSRF (auth a token, non a cookie), gate sul metodo HTTP, risoluzione utente dal Bearer token.
3 · Vista
Chiamata come view(request, user, …); ogni query è ristretta al client_profile dell'utente.
4 · Serializzazione
Builder di dict semplici (_coach_dict, _day_dict, …) → JsonResponse.
Panoramica

Modello di sicurezza

Postura difensiva ad alto livello. I segreti e i dettagli operativi restano fuori dal repo.

Autenticazione

Token firmati con django.core.signing (HMAC su SECRET_KEY): il payload — solo lo uid — è leggibile ma non falsificabile. Validità 30 giorni, inviato come Authorization: Bearer. Sull'app è custodito nel Keychain.

Autorizzazione

Ogni endpoint restringe le query al client_profile dell'utente autenticato. Gli accessi per-id (conversazioni, check, notifiche) verificano la proprietà → niente IDOR.

Credenziali

Password con hash via Django (check_password, confronto a tempo costante). Il login è rate-limited per IP e per account (finestra a bucket in cache), con reset al primo accesso valido.

Trasporto & input

In produzione: HTTPS forzato, HSTS, cookie Secure. Upload immagini ri-codificati server-side (neutralizza payload nascosti), con tetti di dimensione e allowlist MIME. ATS sull'app forza TLS verso i domini reali.

Nota. Il report di audit dettagliato (debolezze residue, vettori, mitigazioni interne) è mantenuto separatamente e non è incluso in questa documentazione pubblica: documentare le debolezze equivale a fornire una mappa all'attaccante.

Backend · config/api.py

Auth & Token

Infrastruttura di autenticazione. Token stateless firmati — nessun modello/migrazione aggiuntiva: il token è un blob firmato che porta l'id utente.

issue_token(user)
Firma {'uid': user.id} con salt dedicato → stringa token. Emesso al login.
_user_from_token(token)
Verifica firma e scadenza (30 gg), ritorna l'User attivo o None.
_bearer(request)
Estrae il token dall'header Authorization: Bearer <token>.
@api_view(methods)
Decoratore: esenzione CSRF + gate sul metodo + auth a token. Inietta user nella vista, o ritorna 401.
# Pattern di ogni endpoint protetto
@api_view(['GET'])
def endpoint(request, user):       # user già autenticato
    client = getattr(user, 'client_profile', None)
    if not client:
        return JsonResponse({'error': …}, status=403)
    # … query SEMPRE filtrata per `client` …

API · /api/v1

Reference completo dei 35 endpoint. Legenda: GET POST PATCH DELETE · 🔒 auth richiede token · pubblico senza token.

Autenticazione

POST/api/v1/auth/loginpubblicologin()

Verifica email + password, emette il token. Rate-limited per IP e account (429 se superato). 401 credenziali errate, 403 email non verificata. Ritorna {token, user}.

POST/api/v1/auth/forgot-passwordpubblicoforgot_password()

Avvia il reset password inviando l'email di recupero. Risposta generica (no enumeration).

Account & Profilo

GET/api/v1/me🔒 authme()

Identità dell'utente + i professionisti collegati (coach / trainer / nutrizionista) + sintesi del profilo (altezza, obiettivo, livello attività, foto).

GETPATCH/api/v1/profile🔒 authprofile()

Legge / modifica il ClientProfile dell'atleta: nome, telefono, altezza, peso, sport, obiettivo, livello, genere, data di nascita.

POST/api/v1/profile/photo🔒 authprofile_photo()

Upload avatar (multipart). L'immagine viene ri-codificata in WebP server-side; ritorna l'URL assoluto.

GETPATCH/api/v1/settings🔒 authsettings()

Preferenze di notifica email per categoria.

POST/api/v1/tutorial/complete🔒 authtutorial_complete()

Segna come visto il tutorial del mascotte Chiron (mostrato al primo accesso).

DELETE/api/v1/account🔒 authdelete_account()

Disattiva l'account lato server.

POST/api/v1/devices/register🔒 authregister_device()

Registra/aggiorna il device token APNs dell'utente per le push notification.

Allenamento

GET/api/v1/workouts🔒 authworkouts()

Piani di allenamento attivi assegnati, con giorni ed esercizi (prescrizione completa: set, rep, carico, recupero, tempo, note).

GET/api/v1/workout-history🔒 authworkout_history()

Sessioni di allenamento passate/completate (storico).

POST/api/v1/sessions/start🔒 authsession_start()

Avvia una WorkoutSession live per un giorno di un piano assegnato.

POST/api/v1/sessions/{id}/log-set🔒 authsession_log_set()

Registra/aggiorna un set (reps, carico, RPE, completato) nella sessione — verifica che la sessione sia dell'utente.

POST/api/v1/sessions/{id}/finish🔒 authsession_finish()

Chiude la sessione (completata o interrotta), ricalcola durata e RPE medio.

Nutrizione

GET/api/v1/nutrition🔒 authnutrition()

Piani nutrizionali attivi: giorni, pasti e alimenti con macro.

GET/api/v1/nutrition/foods🔒 authfood_search()

Ricerca nel database alimenti (param ?q=), ordinata per genericità.

GET/api/v1/nutrition/assignments/{id}/macro-day🔒 authmacro_day()

Modalità MACRO: alimenti loggati oggi + totali consumati + target macro del giorno.

POST/api/v1/nutrition/assignments/{id}/macro-log🔒 authmacro_log_create()

Registra un alimento consumato (assegnazione verificata come MACRO e dell'utente).

DELETE/api/v1/nutrition/macro-log/{entry_id}🔒 authmacro_log_delete()

Elimina una voce del diario macro dell'utente.

GET/api/v1/supplements🔒 authsupplements()

Protocollo integratori attivo dell'atleta.

Check & Progressi

GET/api/v1/checks🔒 authchecks()

Check-in in sospeso (timeline di monitoraggio).

POST/api/v1/checks/{instance_id}/submit🔒 authcheck_submit()

Invia un check in sospeso (peso, misure, note). Rifiuta check già compilati o scaduti.

GET/api/v1/progress🔒 authprogress()

Storico dei check inviati: serie peso, misure, foto, feedback del coach.

GET/api/v1/anamnesis🔒 authanamnesis()

Questionario anamnesi (modulo d'ingresso) in sola lettura.

Agenda & Percorso

GET/api/v1/appointments🔒 authappointments()

Appuntamenti dell'atleta con i coach (tipo, orario, luogo, link meeting, stato).

GET/api/v1/journey🔒 authjourney()

Timeline del percorso: piani, diete ed eventi assegnati. Riusa il builder di eventi del web per tenere allineate vista coach e atleta.

Chat & Notifiche

GET/api/v1/conversations🔒 authconversations()

Elenco conversazioni dell'atleta con ultimo messaggio.

GET/api/v1/conversations/{id}/messages🔒 authmessages()

Messaggi di una conversazione — caricati solo se appartiene all'utente (blocca IDOR).

POST/api/v1/conversations/{id}/send🔒 authsend_message()

Invia un messaggio di testo nella conversazione (proprietà verificata).

GET/api/v1/notifications🔒 authnotifications()

Feed completo delle notifiche dell'utente.

POST/api/v1/notifications/{id}/read🔒 authnotification_read()

Segna come letta una notifica (verifica target_user).

Abbonamento

GET/api/v1/subscription🔒 authsubscription()

Abbonamento attivo dell'atleta: stato, pagamento, date, piano e prezzo.

GET/api/v1/plans🔒 authplans()

Piani di abbonamento offerti dai coach dell'atleta ("Piani e Prezzi").

GET/api/v1/coaches/{id}🔒 authcoach_detail()

Profilo completo di un coach — solo se realmente collegato all'atleta (relazioni/assegnazioni).
